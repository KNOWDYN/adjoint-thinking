# Synthesis Wall

## Official name

Synthesis Wall

## Status

Source-grounded KBC prompt asset. Browser-only. No-code. Platform portable.

## Core reasoning operation

Integrative placement and structure testing.

## Purpose

Turn fragments into defensible structure without hiding provenance, status, contradiction, or human judgment.

## Input definition slot

Provide: fragments, claims, sources, contradictions, boundaries, machine syntheses, desired work product, audience, and pressure.

## Evidence rule

Do not turn unequal materials into smooth prose before material type, trust status, and placement are visible. Machine synthesis is a candidate structure, not evidence.

## Ordered operations

1. Name the structural pressure of the work.
2. Create a Material Table: source claim, boundary, contradiction, machine synthesis, human judgment, or unknown.
3. Create a Status Table: unverified, orientation, working, verified for current use, rejected/rewritten.
4. Create a Placement Table: evidence base, central tension, methods/limits, planning layer, conclusion, appendix, or discard.
5. Identify false-synthesis risks: excessive smoothness, missing tension, equal weight, source amnesia, premature naming, relief mistaken for insight.
6. Propose structure candidates that preserve seams.
7. Choose a provisional architecture and list what must be checked before prose.

## Hard gate

Do not produce final prose if material status and placement are unclear. If the proposed structure hides contradiction, demotes evidence, or lets machine synthesis become authority, reject or rebuild the structure.

## Native artefacts

- Structural Pressure Statement
- Material Table
- Status Table
- Placement Table
- False-Synthesis Risk List
- Provisional Architecture
- Pre-Prose Check List

## Failure and abstention conditions

- The user provides only a topic, not fragments.
- Fragments lack provenance or trust status.
- The machine outline becomes the hidden architecture without inspection.
- Contradictions are smoothed into decorative caveats.

## Unified Response Template reporting clause

Report using `URT_UNIFIED_RESPONSE_TEMPLATE.md` unless the user explicitly asks for another order. The URT controls presentation order only. Do not let the URT change this asset's purpose, evidence rule, hard gate, native artefacts, or decision logic.

## Canonical prompt text

```text
Build a Synthesis Wall for these fragments: [FRAGMENTS]. Desired work product: [WORK PRODUCT]. Audience: [AUDIENCE]. First name the structural pressure. Then create Material, Status, and Placement tables. Identify false-synthesis risks, propose a provisional architecture that preserves seams, and list what must be checked before prose. Do not produce final prose unless the wall supports it.
```

## Short stress-test checklist

- [ ] Does it sort material before prose?
- [ ] Does it assign status separately from material type?
- [ ] Does it preserve contradictions?
- [ ] Does it avoid premature naming?
- [ ] Does it make structure serve pressure, not machine neatness?


## Reasoning-model safety clause

Do not ask for or expose private chain-of-thought. Produce only visible artefacts, concise decision rationales, assumptions, evidence status, source paths, boundaries, and decision-relevant findings. More model reasoning does not upgrade trust status. Trust is upgraded only by provenance, source support, calculation, contradiction search, boundary preservation, domain review, reality contact, or human judgment.
